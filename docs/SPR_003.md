# SPR-003 — Face & Edge Selection + Pivot Set

**Sprint Goal:** Introduce core geometric selection (faces and edges) and enable fast viewport re-centering by setting a rotational pivot with a middle-mouse double click.

**Milestone:** Face / Edge Selection  
**Status:** Planned

---

## Background

With the 3D viewer and toolbar in place, the next usability milestone is direct model interaction. Users need to select individual faces and edges for future operations, and quickly set the orbit pivot to inspect details from a chosen point. This sprint establishes robust picking and visual feedback as the base for upcoming modelling tools.

---

## Tasks

### 1. Selection Data & State
- [x] Define selectable primitive types (`Face`, `Edge`) in renderer/application state
- [x] Add persistent selection state for hovered and active primitives
- [x] Support single active selection at minimum (with optional multi-select deferred)

### 2. Picking / Hit Testing Pipeline
- [x] Implement mouse-picking from screen coordinates into world-space ray
- [x] Add face hit testing against current visible geometry
- [x] Add edge hit testing with distance threshold in screen or world space
- [x] Resolve nearest valid hit when both face and edge are candidates

### 3. Interaction Mapping
- [x] On left click, select hovered face or edge
- [x] Clear active selection when clicking empty viewport space
- [x] Prevent accidental selection while camera drag is in progress

### 4. Visual Feedback
- [x] Highlight hovered primitive distinctly from selected primitive
- [x] Render selected face with clear fill/outline treatment
- [x] Render selected edge with strong contrast/thickness
- [x] Keep highlight rendering stable while orbiting, panning, and zooming

### 5. Pivot Placement (Middle-Mouse Double Click)
- [x] Detect middle-mouse double-click in the viewport
- [x] On valid hit, set orbit camera `target` (rotational pivot) to the clicked point
- [x] Prefer exact hit position on selected geometry; fall back to nearest surface hit
- [x] Add lightweight visual indicator for newly set pivot point (short-lived marker is acceptable)

### 6. Camera Integration
- [x] Ensure subsequent orbit rotation occurs around the updated pivot target
- [x] Preserve current camera distance/orientation when pivot changes
- [x] Keep pan/zoom behavior consistent after pivot updates

### 7. Manual Testing
- [ ] Select multiple faces and confirm correct highlight changes per click
- [ ] Select edges near dense geometry and confirm nearest-edge resolution is reliable
- [ ] Click empty space and confirm selection clears without side effects
- [ ] Middle-mouse double-click on geometry and confirm pivot relocates correctly
- [ ] Orbit immediately after pivot set and confirm rotation centers on new target

## Acceptance Criteria

| # | Criterion |
|---|-----------|
| 1 | Faces and edges can be individually selected with left click |
| 2 | Hover and selected states are visually distinct and stable |
| 3 | Clicking empty viewport space clears the active selection |

| 5 | Orbit rotation uses the updated pivot immediately without camera jumps |
| 6 | No crashes or severe frame drops during normal selection/pivot interaction |

---

## Technical Notes

- **Picking:** Use ray-cast based hit testing from camera + cursor position; keep logic independent from UI widgets
- **Priority rules:** Deterministic tie-breaker required when face and edge hits overlap (e.g. nearest depth first)
- **Tolerance:** Edge hit threshold should be configurable to balance precision and usability
- **Pivot behavior:** Middle-mouse double-click updates `OrbitCamera::target` only; avoid implicit reset of azimuth/elevation/distance
- **Rendering path:** Selection/hover overlays should be drawn as a dedicated pass to avoid z-fighting and style drift
